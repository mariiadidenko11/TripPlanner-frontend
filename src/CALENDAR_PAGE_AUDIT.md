# CalendarPage — аудит логіки даних

Перевірено не візуально, а по ланцюжку «компонент → api.js → бекенд». Нижче — що використовується, що співпадає, і виявлені проблеми/ризики.

---

## 1. Підсумок

Логіка даних **коректна**. Сторінка тягне лише бронювання (reservations) поточного трипа й накладає їх на сітку місяця. Дані трипа (межі поїздки) беруться з outlet-контексту. Назви полів між фронтом і беком узгоджені через нормалізатор `_normalizeBooking`. Сторінка стійка до порожніх масивів і `null`.

Знайдено: 1 потенційний баг середньої важливості (дата-only бронювання) + кілька нюансів. Бракує окремого endpoint для «одного трипа» (зараз обхід через список) — не критично.

---

## 2. Використані API endpoints

| Що | Метод | Endpoint | Хто викликає |
|----|-------|----------|--------------|
| Бронювання трипа | GET | `/api/content/get/reservation/{tripId}` | `getBookings()` → CalendarPage |
| Дані трипа (межі поїздки) | GET | `/api/content/get/trips` → фільтр по id | `getTrip()` → TripLayout (передає в контекст) |
| Завдання (для прогресу в layout) | GET | `/api/content/get/task/{tripId}` | TripLayout (календарю НЕ потрібні) |

CalendarPage напряму викликає **лише `getBookings(tripId)`**. Об'єкт `trip` приходить готовим з `TripLayout` через `useOutletContext()`.

---

## 3. Які дані підтягуються і звідки

### Outlet context (з TripLayout)
```
{ tripId, trip, refetchTrip }
```
- `tripId` — з `useParams()`.
- `trip` — результат `getTrip(tripId)`, нормалізований (`_normalizeTrip`).
- Календар читає з `trip`: `start_at`, `end_at` (межі поїздки для підсвічування днів).

### Бронювання (getBookings → _normalizeBooking)
Кожне бронювання після нормалізації:
```
{ id, name, cost, type, address, note, start_at, end_at, created_at }
```
Календар використовує: `id, name, address, note, cost, start_at` + категорію (`type`/`type_id`).

---

## 4. Відповідність полів backend ↔ frontend

Бек віддає `ShowReservationDTO` (camelCase у JSON). Нормалізатор мапить:

| Backend (ShowReservationDTO) | JSON-поле | Frontend (після _normalizeBooking) | Календар |
|------------------------------|-----------|-------------------------------------|----------|
| `Id` | `id` | `id` | ключ списку |
| `Name` | `name` | `name` | заголовок |
| `Cost` | `cost` | `cost` | вартість у модалці |
| `ReservationType` | `reservationType` | `type` | колір/іконка категорії |
| `Address` | `address` | `address` | підзаголовок |
| `Note` | `note` | `note` | примітка |
| `StartTime` | `startTime` | `start_at` | **дата події в календарі** |
| `EndTime` | `endTime` | `end_at` | (не використовується календарем) |
| `CreatedAt` | `createdAt` | `created_at` | — |

Категорія: `bookingType(b) = b.type ?? b.type_id ?? 'other'`. `type` — з реального беку, `type_id` — з мок-даних. Жодного перейменування/підміни значення.

---

## 5. Моделі / структури

- **Backend:** `ShowReservationDTO[]` (масив). Use case `ExecuteShowReservation(id)` повертає масив, перевіряє власника трипа (`CheckTripOwner`).
- **Frontend нормалізована модель booking:** див. розділ 3.
- **Категорії (лише оформлення):** мапа `BOOKING_CATS` — ключі/кольори/іконки 1:1 з `CATS` на `BookingPage` (`accommodation, transport, food, leisure, shopping, other`) + синоніми (`hotel, flight, restaurant, activity`). Значення `type` НЕ змінюється.
- **Внутрішня структура календаря:** `bookingsByDay` — об'єкт `{ 'YYYY-MM-DD': [event,...] }`, зібраний у `useMemo`.

---

## 6. Обробка масивів / null / порожніх даних

Перевірено — сторінка не падає:
- `getBookings` повертає `(list ?? []).map(...)` → завжди масив.
- `const bookings = bookingsData ?? []` → захист від `null` під час завантаження.
- `if (!b.start_at) return` → бронювання без дати пропускається (не ламає сітку).
- `bookingsForKey(key) = bookingsByDay[key] ?? []` → відсутній день = порожній масив.
- `trip?.start_at` / optional chaining скрізь → якщо `trip === null`, `isTripDay` повертає `false`, календар показує поточний місяць.
- Категорія невідомого типу → fallback на оригінальну назву з нейтральним кольором (не падає).

---

## 7. Залежності між trips / tasks / bookings / places

- **trip → calendar:** межі поїздки (`start_at`/`end_at`) підсвічують діапазон днів. Залежність на один трип (`tripId`).
- **bookings → calendar:** головне джерело подій. Прив'язка по `tripId`, фільтр по наявності `start_at`.
- **tasks:** НЕ використовуються (за вимогою — лише бронювання). Завантажуються в `TripLayout` тільки для прогрес-бару, календарю не передаються.
- **places:** НЕ використовуються календарем.
- Немає крос-залежностей booking↔task↔place у самому календарі — він читає виключно bookings + межі trip.

---

## 8. Виявлені проблеми та ризики

### ⚠️ P1 (середній). Бронювання «тільки з датою, без часу» завжди має дату на беку
`ShowReservationDTO.StartTime` — **не nullable** (`DateTimeOffset`, не `DateTimeOffset?`). Тобто з реального беку кожне бронювання приходить з якоюсь датою початку. Правило «без дати не показувати» спрацює лише для мок-даних (де є `start_at: null`) або якщо бек поверне дефолтну дату `0001-01-01`. 
**Ризик:** якщо бек для «недатованого» бронювання збереже дефолт `0001-01-01`, воно з'явиться в січні 1-го року, а не сховається. 
**Рекомендація:** або зробити `StartTime` nullable на беку (модель `AddReservationModel` уже nullable — `DateTimeOffset?` — а DTO відповіді ні), або на фронті відсікати дефолтну дату: `if (new Date(b.start_at).getFullYear() < 2000) return;`.

### ⚠️ P2 (низький). `getTrip` тягне весь список трипів
Окремого endpoint «один трип за id» на беку немає. `getTrip()` робить `GET /api/content/get/trips` і фільтрує на клієнті. Для календаря це працює, але це зайвий трафік. 
**Рекомендація (не блокер):** додати на бек `GET /api/content/get/trip/{id}`.

### ⚠️ P3 (низький). Часовий пояс дати
Бек віддає `DateTimeOffset` у UTC (`...Z`). Календар групує по **локальній** даті (`toKey` бере локальні рік/місяць/день). Це навмисно (щоб уникнути зсуву від `toISOString`), але для користувачів у часовому поясі, далекому від UTC, подія о 23:00–01:00 може потрапити в сусідній день. 
**Статус:** прийнятно для більшості випадків; якщо потрібна точність — узгодити TZ із беком.

### ℹ️ P4 (інфо). Розбіжність `type` vs `type_id`
Реальний бек використовує `reservationType` → нормалізується в `type`. Мок-дані використовують `type_id`. Календар читає обидва (`b.type ?? b.type_id`), тож працює в обох режимах. На реальному беку поле `type_id` не існує — це лише сумісність з моком.

### ℹ️ P5 (інфо). Категорія `activity` в мок-даних
У мок-даних є `type_id: 'activity'`, якого немає в базовому `CATS`. У `BOOKING_CATS` додано синонім `activity` (вигляд як leisure), тож колір/іконка коректні. На реальному беку значення категорій залежить від того, що збереже користувач.

---

## 9. Чого бракує (endpoints)

- **`GET /api/content/get/trip/{id}`** — окремий трип. Зараз обхід через повний список (P2).
- Для календаря більше нічого не бракує: бронювання, межі трипа — все є.

---

## 10. Висновок

Логіка даних сторінки **узгоджена з беком**, поля співпадають, null/порожні дані оброблені, сторінка не ламається. Єдине, що варто виправити по суті даних — **P1** (nullable `StartTime` у DTO відповіді або відсікання дефолтної дати на фронті). Решта — оптимізації й нюанси, не блокери.
