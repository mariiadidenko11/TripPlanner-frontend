# Trip CalendarPage — Технічна специфікація бекенду

> Файл: `src/pages/trip/CalendarPage.jsx`
> Дата: 30.05.2026

---

## 1. Загальний опис

`Trip CalendarPage` — календар усередині конкретної поїздки. На відміну від глобального `CalendarPage`, він показує **лише бронювання однієї поїздки** та підсвічує дні, що входять у її діапазон дат.

Компонент отримує контекст поїздки (`trip`, `tripId`) від батьківського `TripLayout` через `useOutletContext` — тобто зайвого запиту за даними самої поїздки **не робить**.

**Що відображається:**

- **Дні поїздки** — клітинки від `trip.start_at` до `trip.end_at` підсвічуються фірмовим синім (`rgba(64,179,224,0.16)`). Не клікабельні, якщо немає бронювань.
- **Бронювання** — кольорові чіпи з іконкою категорії на день `startTime`. При кліку на день відкривається модальне вікно з деталями.

## 2. API-ендпоінти

Використовується **один ендпоінт**. Дані поїздки (назва, дати) беруться з контексту — окремого запиту не потрібно.

| Метод | URL | Опис | Доступ |
|-------|-----|------|--------|
| `GET` | `/api/content/get/reservation/{id}` | Всі бронювання для поїздки з `id` | Авторизований, власник поїздки |

### Деталі виклику

Фронтенд (`api.js`):

```js
const { data: bookingsData } = useApi(() => getBookings(tripId), [tripId]);
```

Запит виконується при монтуванні та повторюється при зміні `tripId`. Результат — масив бронювань, нормалізований через `_normalizeBooking`.

## 3. Структури даних (DTO)

### 3.1. ShowReservationDTO — відповідь `/api/content/get/reservation/{id}`

| Поле DTO (C#) | Тип | JSON ключ | Використання в компоненті |
|---------------|-----|-----------|--------------------------|
| `Id` | `int` | `id` | Ключ елементу у списку |
| `Name` | `string` | `name` | Заголовок чіпа та модального вікна (`b.name`) |
| `Cost` | `int` | `cost` | Відображається у модалці: `$1 200` |
| `ReservationType` | `string` | `reservationType` | Визначає іконку, колір чіпа та підпис категорії |
| `Address` | `string` | `address` | Підзаголовок у модалці. Якщо `name` відсутній — використовується як заголовок |
| `Note` | `string` | `note` | Курсивний текст у модалці |
| `StartTime` | `DateTimeOffset` | `startTime` | **Головне поле.** Визначає день на календарі та час у модалці |
| `EndTime` | `DateTimeOffset` | `endTime` | Приймається, але наразі не відображається |
| `CreatedAt` | `DateTimeOffset` | `createdAt` | Приймається, але не відображається |

### 3.2. Нормалізація на фронтенді (`_normalizeBooking`)

```js
{
  id:        r.id,
  name:      r.name ?? '',
  cost:      r.cost ?? 0,
  type:      r.reservationType ?? r.type ?? 'other',   // ← читає reservationType
  address:   r.address ?? '',
  note:      r.note ?? '',
  start_at:  r.startTime ?? r.start_at ?? null,        // ← читає startTime
  end_at:    r.endTime   ?? r.end_at   ?? null,
  created_at: r.createdAt ?? null,
}
```

> ⚠️ **Важливо:** фронтенд читає `r.reservationType` (PascalCase) та `r.startTime` / `r.endTime` — саме так їх повертає `ShowReservationDTO`. Якщо імена полів зміняться — нормалізація зламається.

### 3.3. Контекст поїздки (з TripLayout, без запиту)

Поля об'єкта `trip`, які використовує компонент:

| Поле | JSON ключ бекенду | Використання |
|------|-------------------|--------------|
| `trip.start_at` | `startDate` | Початковий місяць при відкритті (`new Date(trip.start_at)`) |
| `trip.end_at` | `endDate` | Підсвітка діапазону днів поїздки (`isTripDay`) |

Ці поля вже приходять через `GET /api/content/get/trips` і нормалізуються в `_normalizeTrip`:

```js
start_at: t.startDate ?? null,
end_at:   t.endDate   ?? null,
```

> Якщо `startDate` або `endDate` відсутні або `null` — підсвітка діапазону вимикається, а календар відкривається на поточному місяці.

## 4. Логіка відображення та залежності від даних

### 4.1. Визначення дня бронювання

```js
const key = toKey(b.start_at);  // → "YYYY-MM-DD" (локальний час, без UTC-зсуву)
```

Бронювання без `startTime` (`start_at = null`) **не відображаються** на календарі.

### 4.2. Категорії бронювань (ReservationType)

Фронтенд маппить `reservationType` → іконка + колір. Підтримуються такі значення:

| Значення `ReservationType` | Підпис | Іконка | Колір |
|----------------------------|--------|--------|-------|
| `"accommodation"` / `"hotel"` | Проживання | 🏨 | `#8B5CF6` |
| `"transport"` / `"flight"` | Транспорт | ✈️ | `#0EA5E9` |
| `"food"` / `"restaurant"` | Харчування | 🍽️ | `#F97316` |
| `"leisure"` / `"activity"` | Дозвілля | 🎭 | `#EC4899` |
| `"shopping"` | Шопінг | 🛍️ | `#EF4444` |
| `"other"` (fallback) | Інше | 📌 | `#9CA3AF` |

> Якщо значення не збігається з жодним із наведених — фронтенд показує оригінальну назву з типу як підпис та нейтральний колір `📌 #9CA3AF`.

> ⚠️ **Відмінність від глобального CalendarPage:** тут підтримуються тільки англійські ключі (`accommodation`, `transport` тощо). Українські варіанти (`"Готель"`, `"Літак"`) **не маппляться** — вони потраплять у fallback. Рекомендується використовувати англійські значення або синхронізувати обидва компоненти.

### 4.3. Акцентний колір дня

Колір підсвітки клітинки дня визначається за першим бронюванням цього дня:

```js
const dayAccent = has
    ? catMeta(dayBookings[0].type).color   // колір першого бронювання
    : (tripActive ? '#40B3E0' : null);     // або фірмовий primary для днів поїздки
```

### 4.4. Формат дат

```
2025-07-14T15:00:00+02:00   // startTime / endTime бронювань (DateTimeOffset)
2025-07-10T00:00:00Z        // startDate / endDate поїздки (DateTime)
```

Час із `startTime` відображається у модальному вікні:

```js
formatTime(b.start_at)
// → "15:00" (toLocaleTimeString 'uk-UA', hour: '2-digit', minute: '2-digit')
```

## 5. Відмінності від глобального CalendarPage

| Аспект | `app/CalendarPage` | `trip/CalendarPage` |
|--------|--------------------|---------------------|
| Scope | Всі поїздки користувача | Одна поїздка |
| Запити до бекенду | `getTrips()` + `getBookings(id)` для кожної поїздки | `getBookings(tripId)` — один запит |
| Дані поїздки | З API | З контексту `TripLayout` (вже завантажені) |
| Смуга подорожі | Є (багаторядкова, з `tripSegmentsByWeek`) | Немає смуги — лише підсвітка клітинок |
| Picker місяця/року | Є | Немає (лише `←` / `→`) |
| Перехід при кліку | `/trips/{id}/overview` або `/trips/{id}/booking` | Тільки модалка — переходів немає |
| Синоніми ReservationType | Українські + англійські | Тільки англійські |
| Ліміт чіпів на день | 2 | 3 |

---

## 6. Відсутні поля — що може зламати відображення

| Поле | DTO | Наслідок відсутності |
|------|-----|----------------------|
| `StartTime` | `ShowReservationDTO` | Бронювання не відображається на календарі |
| `ReservationType` | `ShowReservationDTO` | Застосовується fallback `📌 #9CA3AF` (не критично) |
| `StartDate` | `GetTripsDTO` | Підсвітка днів поїздки вимикається, календар відкривається на поточному місяці |
| `EndDate` | `GetTripsDTO` | Підсвітка днів поїздки вимикається |

## 8. Чекліст для розробника

- [ ] `GET /api/content/get/reservation/{id}` повертає `startTime` у ISO 8601 з часовою зоною
- [ ] `reservationType` — одне з англійських значень таблиці розділу 4.2 (або `"other"`)
- [ ] `GET /api/content/get/trips` повертає `startDate` та `endDate` для підсвітки діапазону
- [ ] Бронювання без `startTime` не повертаються або повертаються з явним `null`
- [ ] `name` або `address` заповнені — інакше чіп відображатиметься як назва категорії
